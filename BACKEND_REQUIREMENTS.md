## 1. System Overview

BYOC (Bring Your Own Cloud) is a **workspace-scoped, multi-tenant file management platform**. Users connect their own S3-compatible storage buckets; the backend acts as a control plane — it never stores files directly but manages credentials, metadata, permissions, presigned URLs, and all business logic.

### Key architectural invariants

- Every resource (file, folder, share link, activity, notification) belongs to exactly one **workspace**.
- A user may belong to multiple workspaces with different roles in each.
- Storage operations (upload, download, delete) are performed **browser → cloud storage** using presigned URLs generated by the backend. The backend never proxies file bytes.
- Storage provider credentials are **encrypted at rest** and never returned to the frontend in plaintext.
- All workspace-scoped actions must first verify the caller's **membership and role** in that workspace.

## 3. Authentication & Session Management

### 3.1 Google OAuth Flow

**Endpoints needed:**
- `GET /auth/google` — redirect to Google OAuth consent screen
- `GET /auth/google/callback` — handle OAuth callback
  - Exchange code for tokens
  - Upsert user record (create if new, update `googleId` if existing account with same email)
  - Create a `Session` record
  - Set a secure, httpOnly cookie (or return a JWT in response body for SPA usage)
  - Redirect:
    - New user (no workspace) → `/onboard`
    - Returning user → `/app`

### 3.2 Email + Password (future-proofing)

The settings page shows password change forms, so email/password auth must be supported even if initially hidden behind Google OAuth:

- `POST /auth/register` — create account with email + password
- `POST /auth/login` — verify password, create session
- `POST /auth/password/change` — authenticated; requires `currentPassword`, `newPassword`

### 3.3 Session Management

- `GET /auth/me` — return current user + their workspaces list (for WorkspaceProvider hydration on app load)
- `POST /auth/logout` — invalidate current session, clear cookie
- `GET /auth/sessions` — list all active sessions for the current user
- `DELETE /auth/sessions/:sessionId` — revoke a specific session
- `DELETE /auth/sessions` — revoke all sessions except current

Each authenticated request must:
1. Validate the session token (cookie or Bearer)
2. Check session expiry
3. Update `session.lastActiveAt`
4. Attach `req.user` and `req.session` to the request

### 3.4 Two-Factor Authentication

- `POST /auth/2fa/setup` — generate TOTP secret, return QR code URI (do not persist until verified)
- `POST /auth/2fa/verify` — verify TOTP code, persist secret, set `twoFactorEnabled = true`
- `POST /auth/2fa/disable` — verify TOTP code or backup code, disable 2FA
- `POST /auth/2fa/challenge` — used during login when 2FA is enabled; verify code before issuing session

### 3.5 API Key Management

- `GET /auth/api-keys` — list user's API keys (prefix, name, createdAt, lastUsedAt only — never the full key)
- `POST /auth/api-keys` — generate new key; return the full key **once** (show-once pattern)
- `DELETE /auth/api-keys/:keyId` — revoke

API key requests must authenticate via `Authorization: Bearer byoc_<key>` header, skip session lookup, use key hash lookup instead.

## 4. Onboarding Flow

### 4.1 Step 1: Provider Selection

No backend call needed — this is purely frontend UI (the list of available vs. "coming soon" providers).

### 4.2 Step 2: Credentials Submission

`POST /onboard/provider`

**Request:**
```json
{
  "providerType": "aws_s3",
  "bucket": "my-bucket",
  "region": "us-east-1",
  "accessKeyId": "AKIAxxx",
  "secretAccessKey": "secretxxx",
  "endpointUrl": null
}
```

**Backend behavior:**
1. Validate all required fields present and non-empty
2. Attempt to connect to the storage provider:
   - Call `HeadBucket` (S3 API) to verify the bucket exists and credentials are valid
   - The backend makes this call server-side using the provided credentials
3. If verification fails → return `422` with specific error (bucket not found, invalid credentials, insufficient permissions, network error)
4. If verification succeeds:
   - Encrypt the credentials (see §23)
   - Create a `StorageProvider` record with `status = Connected`, `lastChecked = now()`
   - Create the user's first `Workspace` (type = Personal, plan = Free) if they don't have one
   - Associate the provider with that workspace
   - Create default `WorkspacePermissions` and `WorkspaceSecurity` rows
5. Return the workspace + provider details (region, bucket, status)

**Response (success):**
```json
{
  "workspace": { "id": "...", "name": "...", "slug": "..." },
  "provider": { "bucket": "my-bucket", "region": "us-east-1", "status": "Connected" }
}
```

### 4.3 Step 3: Verification Confirmation

The frontend polls or awaits the step 2 response — no separate endpoint needed. Redirect to `/app` after step 2 returns success.

## 5. Workspace Architecture

### 5.1 Workspace CRUD

- `GET /workspaces` — list all workspaces where `userId` is a member
- `POST /workspaces` — create new workspace
  - Validate slug uniqueness
  - Validate plan allows creating another workspace (`Free`: max 1, `Pro`: max 3, `Team`: unlimited)
  - Create workspace + WorkspaceMember (role = Owner) + default Permissions + default Security rows
- `GET /workspaces/:workspaceId` — full workspace details (members, storage, permissions, security)
- `PATCH /workspaces/:workspaceId` — update name/slug/type/color
  - Only Owner or Admin can update
  - Slug change: validate new slug is unique
- `DELETE /workspaces/:workspaceId` — delete workspace
  - Only Owner can delete
  - Must confirm with workspace name string
  - Cascade: delete all files (metadata only; optionally attempt to delete S3 objects as background job), folders, share links, activity logs, notifications, members, provider, permissions, security
  - Cannot delete if user's only workspace (? — frontend doesn't enforce this but consider UX)

### 5.2 Workspace Switching

The frontend `WorkspaceProvider` calls a single endpoint on load to get all workspace data:

`GET /workspaces` → returns array of workspaces with members, storage summary, permissions, security.

This is the primary hydration endpoint for the entire dashboard. Must be fast (indexed, consider caching).

### 5.3 Workspace Slug

- Slugs must be: lowercase, alphanumeric + hyphens, 3–50 chars, globally unique
- Validate format on create and update
- Changing slug: all existing deep-links using the old slug are invalidated (no redirect needed per frontend behavior)

### 5.4 Workspace Ownership Transfer

`POST /workspaces/:workspaceId/transfer` — body: `{ newOwnerId: UUID }`
- Only current Owner can call
- Target user must be a member
- Demotes current owner to Admin, promotes target to Owner
- Logs activity

## 6. Role-Based Access Control

### 6.1 Role Hierarchy

```
Owner > Admin > Member > Viewer
```

Numeric comparison: Owner=4, Admin=3, Member=2, Viewer=1

### 6.2 Per-Action Permission Check

Every workspace-scoped action checks:
1. Is the caller a member of this workspace?
2. Does the caller's role meet the minimum role threshold configured in `WorkspacePermissions`?

```
canUpload           → required to create upload sessions and confirm uploads
canCreateFolders    → required to POST /folders
canShareFiles       → required to POST /share-links
canDeleteFiles      → required to DELETE /files/:id and DELETE /folders/:id
canManageBilling    → required to access billing endpoints
```

### 6.3 Admin-Level Actions (always require Admin+)

- Invite members
- Remove members (excluding Owner)
- Change member roles (cannot elevate above own role)
- Change workspace permissions/security settings
- Connect/disconnect/update storage providers
- Access workspace settings

### 6.4 Owner-Only Actions

- Delete workspace
- Transfer ownership
- Remove Admin-role members
- Change billing plan

### 6.5 Permission Check Middleware

Every route under `/workspaces/:workspaceId/` must run:

```
requireWorkspaceMember(workspaceId)      → attaches req.membership
requirePermission(action)                → checks req.membership.role >= WorkspacePermissions[action]
requireRole(minimumRole)                 → checks req.membership.role >= minimumRole
```

## 7. Storage Provider Abstraction

### 7.1 Provider Interface

The backend must implement a provider-agnostic interface. All S3-compatible providers share the same API surface but differ in endpoint, auth style, and quirks:

**Operations the backend must support through this abstraction:**
- `verifyConnection(bucket, creds)` → boolean + error message
- `generateGetPresignedUrl(bucket, key, expirySeconds)` → URL string
- `initiateMultipartUpload(bucket, key, contentType)` → uploadId string
- `generateUploadPartPresignedUrl(bucket, key, uploadId, partNumber, expirySeconds)` → URL string
- `completeMultipartUpload(bucket, key, uploadId, parts[{partNumber, etag}])` → ETag
- `abortMultipartUpload(bucket, key, uploadId)` → void
- `headObject(bucket, key)` → { size, lastModified, contentType, etag }
- `deleteObject(bucket, key)` → void
- `deleteObjects(bucket, keys[])` → void (batch delete)
- `listObjects(bucket, prefix, maxKeys, continuationToken)` → { objects[], nextToken }
- `getRequiredPermissions()` → string[] (the IAM/policy permissions list to show users in UI)

**Provider implementations to build (only aws_s3 active initially; others stubbed):**
- `AwsS3Provider` — standard AWS SDK
- `CloudflareR2Provider` — S3-compatible, no `region` needed
- `BackblazeB2Provider` — S3-compatible endpoint
- `MinioProvider` — S3-compatible, custom endpoint
- `GcsProvider` — (coming soon — different auth)
- `AzureBlobProvider` — (coming soon — different API entirely)

### 7.2 Credential Decryption Guard

Credentials must be decrypted **only** inside the provider abstraction layer, never before. The decrypted credentials must not be logged, serialized to JSON responses, or stored in any cache. They must be held in memory only for the duration of the SDK call.

### 7.3 Provider Health Check

`POST /workspaces/:workspaceId/provider/health-check`

- Decrypt credentials
- Call `HeadBucket` (or equivalent)
- Update `StorageProvider.status` and `StorageProvider.lastChecked`
- Return `{ status, lastChecked, latencyMs }`

This endpoint is called:
- From the dashboard home `ProviderHealthCard` (on mount)
- From the integrations settings page "Re-check connection" button
- From a background cron job (see §20)

## 8. File System & Metadata

### 8.1 File Listing (Files Page)

`GET /workspaces/:workspaceId/files`

**Query parameters:**
```
folderId        string | "root"   (default: root)
kind            all | image | video | document | archive
search          string
page            integer
limit           integer
sortBy          name | size | uploadedAt | lastModified
sortOrder       asc | desc
```

**Response:**
```json
{
  "files": [FileItem...],
  "folders": [FolderNode...],
  "breadcrumb": [{ "id": "...", "name": "..." }...],
  "total": 245,
  "page": 1,
  "limit": 50
}
```

The `breadcrumb` array walks from root to the requested `folderId`, enabling the frontend breadcrumb component.

### 8.2 File Detail

`GET /workspaces/:workspaceId/files/:fileId`

Returns full `FileItem` including:
- All metadata fields
- `activities` array (last 20 activity log entries for this file)
- `shareLink` object if one exists
- A short-lived (60s) presigned GET URL for preview rendering (never cache this URL in the response)

### 8.3 File Rename

`PATCH /workspaces/:workspaceId/files/:fileId`

Body: `{ name: string }` (extension handled separately or preserved automatically)

- Validate name (no path separators, no null bytes, max 255 chars)
- Update `File.name` and `File.lastModified`
- Log activity: `renamed` with `{ before: oldName, after: newName }`
- Note: the actual S3 object key is **not** renamed (copy + delete would be expensive). The display name and the storage path are separate.

### 8.4 File Move

`PATCH /workspaces/:workspaceId/files/:fileId/move`

Body: `{ targetFolderId: string | null }`

- Validate target folder exists and belongs to same workspace
- Update `File.folderId`
- Log activity: `moved`
- S3 object key is NOT changed (same as rename)

### 8.5 File Delete

`DELETE /workspaces/:workspaceId/files/:fileId`

- Check `canDeleteFiles` permission
- Verify no active share links? (or soft-delete all associated share links too)
- Mark `File.status = deleted`
- Enqueue background job to delete the S3 object (don't block the response)
- Soft-delete any associated `ShareLink` records (set status = Disabled)
- Log activity: `deleted`
- Create `Notification` for workspace members if `enableActivityLogs`

### 8.6 File Preview URL

`GET /workspaces/:workspaceId/files/:fileId/preview-url`

- Decrypt provider credentials
- Generate presigned GET URL with 5-minute expiry
- Return `{ url, expiresAt }`
- Never cache this response

### 8.7 File Download URL

`GET /workspaces/:workspaceId/files/:fileId/download-url`

- Same as preview-url but with `Content-Disposition: attachment` hint embedded in the presigned URL (if provider supports response-headers in presigned URLs — AWS S3 does)
- Log activity: `downloaded`

## 9. Production Multipart Upload Architecture

This is the most complex piece. The backend never touches file bytes — it only orchestrates presigned URLs and tracks state.

### 9.1 Upload Flow (Happy Path)

```
Browser                          Backend                          S3
  │                                 │                              │
  │── POST /upload/initiate ────────▶│                              │
  │                                 │── InitiateMultipartUpload ──▶│
  │                                 │◀── uploadId ─────────────────│
  │◀── { sessionId, uploadId,        │                              │
  │      parts[{ partNumber,         │                              │
  │              presignedUrl }] } ──│                              │
  │                                 │                              │
  │── PUT presignedUrl (part 1) ────────────────────────────────────▶│
  │◀── ETag ────────────────────────────────────────────────────────│
  │                                 │                              │
  │── PUT presignedUrl (part N) ────────────────────────────────────▶│
  │◀── ETag ────────────────────────────────────────────────────────│
  │                                 │                              │
  │── POST /upload/:sessionId/      │                              │
  │       complete ─────────────────▶│                              │
  │   body: { parts[{partNum,etag}]}│── CompleteMultipartUpload ──▶│
  │                                 │◀── success ──────────────────│
  │◀── { file: FileItem } ──────────│                              │
```

### 9.2 Initiate Endpoint

`POST /workspaces/:workspaceId/upload/initiate`

**Request:**
```json
{
  "files": [
    {
      "name": "video.mp4",
      "size": 524288000,
      "mimeType": "video/mp4",
      "folderId": "uuid-or-null"
    }
  ],
  "chunkSizeMB": 10
}
```

**Backend behavior (per file):**
1. Check `canUpload` permission
2. Generate `storagePath`: `{workspaceId}/{folderId or 'root'}/{uuid}/{sanitized-filename}`
3. Call `initiateMultipartUpload(bucket, storagePath, mimeType)` → get `providerUploadId`
4. Calculate `totalChunks = ceil(size / chunkSizeMB * 1024 * 1024)`
5. Generate presigned URLs for all parts (batch, not on-demand)
   - Each part URL has 1-hour expiry
   - AWS S3 minimum part size = 5MB (enforce on backend)
   - Last part can be smaller
6. Create `File` record with `status = uploading`
7. Create `UploadSession` record
8. Log activity: no — only log on completion

**Response:**
```json
{
  "sessions": [
    {
      "sessionId": "uuid",
      "fileId": "uuid",
      "fileName": "video.mp4",
      "parts": [
        { "partNumber": 1, "presignedUrl": "https://s3.amazonaws.com/..." },
        { "partNumber": 2, "presignedUrl": "https://s3.amazonaws.com/..." }
      ],
      "expiresAt": "2026-05-10T14:00:00Z"
    }
  ]
}
```

### 9.3 Progress Reporting

`PATCH /workspaces/:workspaceId/upload/:sessionId/progress`

**Request:**
```json
{ "completedParts": [{ "partNumber": 1, "etag": "abc123" }] }
```

- Merge into `UploadSession.completedChunks`
- Useful for resume: if browser closes, on next load the session can be resumed
- Update `UploadSession.status = in_progress`
- No response body needed (204)

### 9.4 Complete Endpoint

`POST /workspaces/:workspaceId/upload/:sessionId/complete`

**Request:**
```json
{
  "parts": [
    { "partNumber": 1, "etag": "\"abc123\"" },
    { "partNumber": 2, "etag": "\"def456\"" }
  ]
}
```

**Backend behavior:**
1. Load `UploadSession`; verify it belongs to this workspace and user
2. Verify all parts are accounted for (`parts.length === session.totalChunks`)
3. Call `completeMultipartUpload(bucket, storagePath, providerUploadId, parts)`
4. Call `headObject(bucket, storagePath)` to get final size and metadata
5. Update `File` record: `status = ready`, `size` (confirmed), `lastModified = now()`
6. Update `UploadSession.status = completed`
7. Log activity: `uploaded`
8. Create notifications for workspace members (if large workspace, batch/async)
9. Return full `FileItem`

### 9.5 Abort Endpoint

`POST /workspaces/:workspaceId/upload/:sessionId/abort`

1. Call `abortMultipartUpload` on the provider
2. Update `UploadSession.status = aborted`
3. Delete the `File` record (or set `status = failed`)
4. Return 204

### 9.6 Resume Flow

`GET /workspaces/:workspaceId/upload/:sessionId`

- Return current session state: `{ status, completedParts, totalChunks }`
- Frontend can regenerate missing presigned URLs or request fresh ones

`POST /workspaces/:workspaceId/upload/:sessionId/refresh-urls`

- Re-generate presigned URLs for incomplete parts only
- Required when session's original presigned URLs have expired (>1 hour since initiation)

### 9.7 Session Expiry & Cleanup

- Upload sessions expire after 24 hours if not completed
- Background job (see §20) runs every hour:
  - Find sessions where `expiresAt < now()` and `status NOT IN (completed, aborted)`
  - Call `abortMultipartUpload` for each (to avoid orphan S3 multipart uploads which incur storage costs)
  - Set `status = aborted`, `File.status = failed`

### 9.8 Concurrent Uploads

The frontend sends multiple files in one `initiate` call. The backend must handle this atomically or return partial successes. Recommended: create all sessions, return all, let browser upload in parallel.

### 9.9 Single-Part Upload (Small Files)

For files under 5MB (S3 minimum multipart size), use a single presigned PUT URL instead of multipart:

`POST /workspaces/:workspaceId/upload/presign`

```json
{
  "name": "logo.png",
  "size": 45000,
  "mimeType": "image/png",
  "folderId": null
}
```

Returns: `{ fileId, presignedPutUrl, expiresAt }`

Then: `POST /workspaces/:workspaceId/upload/:fileId/confirm` to finalize metadata.

### 11.1 Create Share Link

```json
  "password": "hunter2",
```

**Backend behavior:**
1. Check `canShareFiles` permission
2. Check workspace security: if `disablePublicSharing` and `accessType === Public` → reject with 403

5. Hash password if provided (bcrypt)
6. Generate unique slug (8–12 random URL-safe chars; retry if collision)
7. Create `ShareLink` record
8. If file had no previous share link, update `File.shareLink`
9. Log activity: `shared`
10. Create `link_created` notification for file owner if not self
11. Return full share link object (with `shareUrl = https://app.byoc.io/s/{slug}`)

```
search    string (searches fileName)
page, limit
```

**Response includes stats:**

```json
    "passwordProtectedCount": 4
  }
}
```

### 11.3 Get Share Link Details

`GET /workspaces/:workspaceId/share-links/:linkId`

### 11.4 Update Share Link

`PATCH /workspaces/:workspaceId/share-links/:linkId`

Accepts any subset of: `{ accessType, password, expiresAt, allowDownload, status }`.

- Changing `accessType` re-validates workspace security rules

### 11.5 Delete Share Link

`DELETE /workspaces/:workspaceId/share-links/:linkId`

- Delete `ShareLink` record and all `ShareLinkVisit` records
- Update `File.shareLink = null` if this was the file's associated link
- Log activity: `unshared`

### 11.6 Public Share Link Access (unauthenticated route)

`GET /s/:slug` — public endpoint (no auth)

1. Look up `ShareLink` by slug
2. Check `status === Active`
3. Check `expiresAt` — if expired, update status to `Expired`, return 410 Gone
4. If `accessType === Private` → return 403 (must be invited user — see §11.7)
5. If `accessType === PasswordProtected` and no `X-Share-Password` header → return 401 with `{ requiresPassword: true }`
6. If password provided → verify hash
7. Record `ShareLinkVisit` (increment `visits` counter, create visit record with hashed IP)
8. Return: `{ fileName, fileType, allowDownload, previewUrl?, downloadUrl? }`
   - `previewUrl`: short-lived presigned GET URL (2 min)
   - `downloadUrl`: only if `allowDownload === true`, presigned GET URL with Content-Disposition

### 11.7 Private Share Link (Invite-based)

When `accessType === Private`:
- The link owner must explicitly share it with specific email addresses
- `POST /workspaces/:workspaceId/share-links/:linkId/invite` — body: `{ emails: string[] }`
- The backend validates the visitor's authenticated session before serving content
- This requires `allowPrivateInviteSharing` security setting to be `true`

At upload completion time, the backend must extract and persist:

## 19. Security Hardening Requirements

### 19.1 Authentication Security

- All session tokens must be cryptographically random (minimum 32 bytes from CSPRNG)
- Session tokens must be stored as bcrypt hashes in the database
- Tokens transmitted via httpOnly, Secure, SameSite=Strict cookies
- Cookie expiry: 7 days rolling (refreshed on each request within 1-day window)
- API keys transmitted via Authorization header only (never in cookies)

### 19.2 Input Validation

Every endpoint must validate:
- All required fields present and correct type
- String length limits enforced
- No path traversal in file names or folder names (`../`, null bytes, etc.)
- File extension vs. MIME type consistency check (detect MIME spoofing)
- UUIDs validated as proper v4 UUIDs before database lookup
- Enum values validated against allowed set

### 19.3 Workspace Isolation

- Every database query involving workspace-scoped resources must include `WHERE workspaceId = ?` even if the resource ID is globally unique — prevents IDOR vulnerabilities
- Pattern: look up by `(id, workspaceId)` pair, not by `id` alone

### 19.4 Presigned URL Safety

- Presigned URLs must always point to the specific `storagePath` of the resource
- Never generate presigned URLs for arbitrary keys provided by the client
- Upload presigned URLs must include `Content-Type` constraint so the client cannot upload arbitrary content types

### 19.5 Credential Security

- Storage provider credentials must be encrypted using AES-256-GCM with a key stored in environment (not database)
- Encryption key must be rotatable without downtime (store key version with each encrypted blob)
- Never log credentials, presigned URLs, or session tokens
- Mask credentials in error messages and audit logs

### 19.6 CORS

- Restrict `Access-Control-Allow-Origin` to the frontend domain
- Only allow necessary methods per route
- `credentials: true` only for first-party frontend domain

### 19.7 SQL Injection Prevention

- Use parameterized queries or ORM at all times
- Never interpolate user input into SQL strings

### 19.8 File Type Validation

- At upload completion, optionally call `headObject` to verify `Content-Type` matches what was declared
- Reject uploads where MIME type is `text/html`, `application/javascript`, or other executable types unless explicitly allowed

## 20. Background Jobs & Events

The following jobs must run on a schedule or in response to events:

### 20.1 Upload Session Cleanup (every 30 minutes)
- Find sessions where `expiresAt < now()` and `status IN (pending, in_progress)`
- Call `abortMultipartUpload` on provider
- Set `status = aborted`, `File.status = failed`
- Log metric: aborted upload count

### 20.2 Share Link Expiry Checker (every hour)
- Find `ShareLink` where `expiresAt < now()` and `status = Active`
- Batch update `status = Expired`
- Create `link_expired` notifications for link creators (respect notification preferences)

### 20.3 Storage Snapshot (daily, midnight UTC)
- For each workspace with a connected provider:
  - Aggregate file sizes by kind from `File` table
  - Insert row into `StorageSnapshot` table
- Used by analytics trend charts

### 20.4 Provider Health Check (every 6 hours)
- For each connected provider:
  - Call `HeadBucket` using stored credentials
  - Update `status` and `lastChecked`
  - If status changes to Error: create `storage_warning` notification for workspace owner

### 20.5 Storage Warning Alerts (triggered by snapshot job)
- If storage usage crosses 80% of plan limit: create `storage_warning` notification (once per day max)
- If storage crosses 95%: create additional `storage_warning` with higher urgency

### 20.6 Notification Retention Cleanup (daily)
- Delete notifications older than 30 days where `dismissed = true` or `read = true`
- Hard-delete notifications older than 90 days unconditionally

### 20.7 Activity Log Retention (daily)
- Delete `ActivityLog` entries older than plan retention limit:
  - Free: 7 days
  - Pro: 30 days
  - Team: 90 days

### 20.8 Data Export Job (triggered by `POST /users/me/export`)
- Compile user data into JSON archive
- Upload archive to internal BYOC bucket with 24-hour presigned URL
- Send email with download link

### 20.9 Orphaned Object Cleanup (weekly)
- List objects in each workspace's S3 bucket with the BYOC key prefix
- Cross-reference against `File` records with `status = ready`
- Flag any objects not referenced by any file (orphaned by failed uploads, etc.)
- Do NOT auto-delete — surface to workspace owner as a report (or delete after 7-day grace period)

## 22. Caching Strategy

### 22.1 What to Cache (Redis or in-process)

| Data | TTL | Invalidation |
|------|-----|--------------|
| `GET /workspaces` (list for user) | 60s | On workspace create/update/delete, member change |
| `GET /workspaces/:id` (full detail) | 30s | On any mutation |
| `GET /workspaces/:id/dashboard` | 120s | On file upload/delete, provider health change |
| Provider health check result | 5min | On manual health check request |
| Notification unread count | 30s | On new notification, mark-read |
| Billing plan list | 1 hour | Static, rarely changes |
| User preferences | 5min | On preference update |
| Share link slug lookup | 60s | On link status change/delete |
| Storage breakdown for analytics | 1 hour | On upload/delete |

### 22.2 What NOT to Cache

- Presigned URLs (they are time-limited by design)
- Decrypted credentials (never persist)
- Session tokens (always verify against DB)
- File listing (too dynamic and workspace-scoped — too complex to invalidate correctly in v1)

### 22.3 Cache Key Strategy

All cache keys must be workspace-scoped to prevent cross-workspace data leakage:
```
workspace:{workspaceId}:dashboard
workspace:{workspaceId}:members
user:{userId}:workspaces
user:{userId}:notifications:count
sharelink:slug:{slug}
```

## 23. Encryption Requirements

### 23.1 Storage Provider Credentials

Algorithm: AES-256-GCM

Each encrypted blob format:
```json
{
  "version": 1,
  "iv": "<base64 12-byte IV>",
  "tag": "<base64 16-byte auth tag>",
  "ciphertext": "<base64 encrypted JSON>"
}
```

The encryption key is loaded from environment variable (`BYOC_CRED_ENCRYPTION_KEY`), never from the database.

The plaintext JSON contains:
```json
{
  "accessKeyId": "AKIAxxx",
  "secretAccessKey": "full-secret",
  "endpointUrl": null
}
```

On retrieval: decrypt in-memory, use for SDK call, discard immediately. Never include in API response.

### 23.2 Key Rotation

Store `version` field in each encrypted blob. When rotating keys:
1. Add new key under new version
2. Background job re-encrypts all blobs with the new key
3. Remove old key once all blobs are migrated

### 23.3 2FA Seeds

TOTP seeds are encrypted using the same AES-256-GCM scheme before being stored in `User.twoFactorSecret`.

### 23.4 Sensitive Fields in Logs

Configure logging library to mask:
- `password`, `secret`, `accessKey`, `secretAccessKey`, `token`, `authorization`, `cookie`
- Any field matching `/cred|secret|key|pass|token|auth/i`

## 24. Rate Limiting & Abuse Prevention

### 24.1 Rate Limit Tiers

| Endpoint Group | Limit | Window | Key |
|---------------|-------|--------|-----|
| `POST /auth/login` | 10 attempts | 15 min | IP + email |
| `POST /auth/register` | 5 attempts | 1 hour | IP |
| `POST /auth/2fa/challenge` | 5 attempts | 15 min | IP + userId |
| `GET /s/:slug` (public share) | 60 requests | 1 min | IP |
| Upload initiate | 20 per hour | 1 hour | userId |
| General API | 300 requests | 1 min | userId |
| Workspace create | 10 per day | 24 hours | userId |

### 24.2 Share Link Anti-Scraping

- Public share link endpoints should verify `User-Agent` is a browser (soft check)
- Log IP hash for abuse tracking
- If a single IP accesses >100 share links per hour, temporarily block IP from public link routes

### 24.3 File Upload Abuse

- Max file size: configurable per plan (e.g., Free: 2GB per file, Pro: 50GB, Team: unlimited)
- Max concurrent upload sessions per workspace: 10
- Max total upload sessions per user per day: 100

### 24.4 Brute Force Protection

- After 5 failed login attempts from same IP: impose 15-minute lockout
- After 10 failed 2FA attempts: lock account, require email verification to unlock, create `security` notification

## 25. Error Handling Contract

### 25.1 Standard Error Response

All errors must return JSON:
```json
{
  "error": {
    "code": "WORKSPACE_NOT_FOUND",
    "message": "The requested workspace does not exist or you do not have access.",
    "details": {}
  }
}
```

`message` is user-facing. `code` is machine-readable. `details` contains field-level validation errors when applicable.

### 25.2 HTTP Status Code Policy

| Status | When to use |
|--------|-------------|
| 200 | Success with body |
| 201 | Resource created |
| 204 | Success with no body |
| 400 | Malformed request (bad JSON, wrong types) |
| 401 | Not authenticated |
| 403 | Authenticated but not authorized (wrong role, plan limit, workspace security policy) |
| 404 | Resource not found (use the same message for resources the user might not have access to, to prevent IDOR enumeration) |
| 409 | Conflict (slug already taken, duplicate invite) |
| 410 | Gone (expired share link) |
| 422 | Validation error (missing fields, format errors) |
| 429 | Rate limited |
| 500 | Internal server error |
| 503 | Provider temporarily unreachable |

### 25.3 Provider Errors

When storage provider operations fail, normalize error types:

| Provider Error | Backend Code | HTTP Status |
|----------------|--------------|-------------|
| Invalid credentials | `PROVIDER_AUTH_ERROR` | 503 |
| Bucket not found | `PROVIDER_BUCKET_NOT_FOUND` | 422 |
| Insufficient permissions | `PROVIDER_PERMISSION_DENIED` | 422 |
| Network timeout | `PROVIDER_TIMEOUT` | 503 |
| Provider unavailable | `PROVIDER_UNAVAILABLE` | 503 |
| Object not found in S3 | `FILE_NOT_IN_STORAGE` | 404 |

### 25.4 Partial Success

For batch operations (multi-file upload initiation, batch deletes), return partial results:
```json
{
  "succeeded": [...],
  "failed": [
    { "item": "...", "error": { "code": "...", "message": "..." } }
  ]
}
```
